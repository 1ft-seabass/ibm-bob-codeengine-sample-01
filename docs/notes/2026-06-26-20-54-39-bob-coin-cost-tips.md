## 問題

1セッションで Bob Coin の消費量が大幅に増加した。原因と対策を整理する。

## 原因分析

### 1. コンテキストが広範囲に及んだ（最大要因）

1つのセッションで以下をすべてカバーした：

- `docs/` 構造導入
- セキュリティチェック導入（secretlint + gitleaks + simple-git-hooks / Phase 0〜3）
- IBM Code Engine 環境セットアップ（ibmcloud CLI・プラグイン・ビルド・デプロイ）
- Bootstrap 適用
- イメージクリーンアップ・トラブル対応
- `dotenv-cli` 導入
- ノート 4本 + 申し送り 1本の作成

扱ったファイルの種類も多く（`package.json`・`.gitignore`・`Dockerfile`・`server.js`・`.env.ce`・`docs/notes/*`・`docs/letters/*`）、毎回のやり取りで AI がそれらを参照・更新し続けた。

### 2. コマンド実行のトライ＆エラーが多かった

- `ibmcloud ce build create` → オプション名の誤り → 修正して再実行
- gitleaks インストール → ZIP ロック競合 → PowerShell で手動インストール
- `application create --build` → 存在しないオプション → `--build-source` に修正
- アプリ応答なし → 原因調査 → イメージ参照修正

1つのゴールに対して複数回コマンドを実行するターンが重なった。

### 3. ファイルの読み書きが多かった

指示書（`doc_note_and_commit.md`・`00_session_end.md`）、テンプレート（`TEMPLATE.md`）、スクリプト（`install-gitleaks.js`・`security-verify.js`）など、長いファイルを何度も読み込む操作がコンテキストを圧迫した。

## 対策・工夫

| 工夫 | 効果 |
|------|------|
| **セッションを小さく区切る**（申し送り運用） | コンテキスト初期化でリセット |
| **1セッション1テーマ**にする | 読み込むファイルを絞れる |
| **トライ＆エラーが予想される作業**（CLI セットアップ等）は事前に手動確認しておく | コマンド往復ターンを減らせる |
| **申し送りを早めに作る**（トークン使用量 75〜85% 目安） | 後半の重いターンを次セッションに回せる |

## 学び

- 「初回セットアップ全部一気にやりきる」セッションは性質上コスト高になりやすい。次回以降は申し送り運用を軌道に乗せることで落ち着く
- CLI ツールのセットアップ（インストール・オプション確認）は特にトライ＆エラーが多くなりやすい。事前に手動で確認しておくか、セッションを分割するとよい
- コンテキストに乗せるファイルは必要最小限にする。長い指示書やテンプレートは毎回全文読まず、差分だけ参照できる形に整理すると効率が上がる
